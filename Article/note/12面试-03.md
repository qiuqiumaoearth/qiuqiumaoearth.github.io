# 一、事件循环核心规则

## 1️⃣ 执行顺序总规则

```
1. 执行一个宏任务（整体 script 也是宏任务）
2. 执行当前产生的所有微任务
3. 可能进行页面渲染
4. 进入下一个宏任务
```

---

## 2️⃣ 宏任务（Macrotask）

常见：

- script（整体代码）
- setTimeout
- setInterval
- I/O
- DOM 事件

---

## 3️⃣ 微任务（Microtask）

常见：

- Promise.then
- Promise.catch
- Promise.finally
- queueMicrotask
- MutationObserver

---

# 四、关键知识点深化

## 1️⃣ 微任务一定在当前宏任务结束后立即清空

```
宏任务执行完 → 清空所有微任务 → 才进入下一个宏任务
```

---

## 2️⃣ 微任务可以在宏任务内部产生

例如：

```js
setTimeout(() => {
  Promise.resolve().then(...)
})
```

这个微任务不会立即执行，而是：

```
等当前 setTimeout 宏任务执行完 → 再执行微任务
```

---

## 3️⃣ Promise 为什么按顺序执行？

因为：

```
微任务队列是先进先出（FIFO）
```

# 一、Promise 执行的本质

## 1️⃣ Promise 构造函数是同步执行的

```js
new Promise(resolve => {
  console.log(1)
  resolve()
})
```

输出：

```
1
```

原因：

```
Promise 构造函数里的 executor 是同步执行
```

---

## 2️⃣ then 是微任务

```js
Promise.resolve().then(() => {
  console.log(2)
})
```

then 回调：

```
会被放入微任务队列
```

---

# 二、重点问题

> 为什么 resolve 会先执行？

答案：

```
resolve() 是同步调用
但 resolve 触发的 then 回调是微任务
```

也就是说：

```
1️⃣ executor 同步执行
2️⃣ resolve() 改变状态
3️⃣ then 回调被加入微任务队列
```

---

# 一、题 3 解析

## 题目

```js
console.log(1)

Promise.resolve()
  .then(() => {
    console.log(2)
    return Promise.resolve(3)
  })
  .then(res => {
    console.log(res)
  })

console.log(4)
```

---

# 第一轮：主线程（宏任务）

执行：

```
console.log(1) → 1
注册微任务（第一个 then）
console.log(4) → 4
```

当前输出：

```
1
4
```

---

# 第二轮：执行微任务

执行第一个 then：

```
console.log(2) → 2
return Promise.resolve(3)
```

⚠ 关键点来了：

```
return Promise.resolve(3)
```

这不会立刻把 3 传给下一个 then。

它会：

1️⃣ 生成一个新的 Promise  
2️⃣ 等这个 Promise resolve  
3️⃣ 才触发下一个 then

但是：

Promise.resolve(3) 是立即 resolved 状态

所以：

它会在当前微任务执行完后，立即注册一个新的微任务。

---

# 微任务队列变化过程

初始：

```
[第一个 then]
```

执行第一个 then 后：

```
[第二个 then]
```

---

# 执行第二个 then

```
console.log(res)
```

这里的 res 是：

```
3
```

---

# 最终输出

```
1
4
2
3
```

# 核心总结

## async 执行规则

```

3️⃣ 当前宏任务执行完后，执行微任务
```
