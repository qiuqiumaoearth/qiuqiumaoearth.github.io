## 一、问题 1

> JS 是单线程，为什么 setTimeout 不阻塞？

你回答：

> 先执行同步任务，再执行任务队列中的异步任务

方向对，但不够底层。

🔥 更底层解释

JavaScript 只有一个主线程（执行栈）。

但浏览器环境提供了：

- Web APIs（定时器、DOM、网络请求等）

当执行：

> setTimeout(...)

流程是：

```txt
1️⃣ 主线程遇到 setTimeout
2️⃣ 交给浏览器的 Web API 处理计时
3️⃣ 主线程继续往下执行
4️⃣ 时间到后，回调被放入任务队列
5️⃣ 等执行栈清空后，再执行回调
```

所以：

> JS 是单线程执行代码，但浏览器是多线程环境。

这句话是面试加分点。

## 二、问题 2

你说：

> 宏任务由浏览器执行微任务由 JS 引擎执行

⚠ 这个说法不准确。

更标准的说法是：

宏任务（Macrotask）

常见：

- setTimeout

- setInterval

- script 整体代码

- I/O

- UI 事件

微任务（Microtask）

常见：

- Promise.then

- MutationObserver

- queueMicrotask

核心区别不是“谁执行”

而是：

**执行顺序不同。**

规则：

```txt
1️⃣ 执行一个宏任务
2️⃣ 执行所有微任务
3️⃣ 渲染页面
4️⃣ 进入下一个宏任务
```

这才是关键。

## 三、问题 3 解析

代码：

```js
console.log(1)

setTimeout(() => {
  console.log(2)
}, 0)

Promise.resolve().then(() => {
  console.log(3)
})

console.log(4)
```

执行流程第一轮：

```txt
主线程（宏任务）console.log(1) → 1
setTimeout → 注册到宏任务队列
Promise.then → 注册到微任务队列
console.log(4) → 4
```

输出：

> 1 4

第二轮：

> 执行微任务console.log(3)

输出：

> 3

第三轮：

> 执行宏任务console.log(2)

输出：

> 2

✅ 最终顺序 1 4 3 2
