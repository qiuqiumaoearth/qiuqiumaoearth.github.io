## 一、JavaScript 执行机制总览

JavaScript 代码执行分为两个阶段：

```txt
1️⃣ 创建阶段（编译阶段）
2️⃣ 执行阶段（运行阶段）
```

1️⃣ 创建阶段会做什么？

在这个阶段，JS 引擎会：

- 创建执行上下文（Execution Context）

- 确定作用域

- 处理变量提升

- 建立作用域链

2️⃣ 执行阶段会做什么？

- 从上到下逐行执行代码

- 进行赋值

- 执行函数调用

## 二、执行上下文（Execution Context）

执行上下文可以理解为：

> 代码运行时的环境

JS 有三种执行上下文：

- 全局执行上下文

- 函数执行上下文

- eval 执行上下文（基本不用）

执行上下文内部包含什么？

一个执行上下文包含：

```txt
1️⃣ 变量环境（Variable Environment）
2️⃣ 作用域链（Scope Chain）
3️⃣ this 绑定
```

## 三、变量提升（Hoisting）

1️⃣ var 的提升

```js
console.log(a)
var a = 1
```

等价于：

```js
var a
console.log(a)
a = 1
```

特点：

- 被提升

- 初始化为 undefined

2️⃣ function 的提升

```js
test()

function test() {
  console.log(1)
}
```

函数声明会整体提升

3️⃣ let / const 的提升

```js
console.log(a)
let a = 1
```

特点：

- 也会提升

- 不会初始化

- 存在暂时性死区（TDZ）

🔥 暂时性死区（TDZ）

- 从作用域开始到变量声明之间的区域。

- 在这个区域访问变量会报错。

## 四、作用域（Scope）

作用域决定变量的可访问范围。

JS 是：

> 词法作用域（Lexical Scope）

也就是说：

- 作用域在写代码的时候就确定

- 不是运行时决定

1️⃣ 全局作用域

在最外层声明的变量。

2️⃣ 函数作用域

```js
function test() {
  var a = 1
}
```

var 受函数控制。

3️⃣ 块级作用域

```js
{
  let a = 1
}
```

let 和 const 受块控制。

## 五、作用域链（Scope Chain）

当访问一个变量时：

> 当前作用域 → 父作用域 → 再往上 → 直到全局

这条查找路径叫：

> 作用域链

示例：

```js
var a = 1

function outer() {
  var b = 2

  function inner() {
    console.log(a)
    console.log(b)
  }

  inner()
}

outer()
```

inner 查找顺序：

```txt
inner 作用域
→ outer 作用域
→ 全局作用域
```

## 六、闭包（Closure）

定义

> 闭包 = 函数 + 该函数能访问的外部变量

更准确的说法：

> 当函数可以访问并持有其词法作用域中的变量时，就形成了闭包。

为什么会产生闭包？

因为：

- 函数执行完 normally 会销毁

- 但如果内部函数被返回

- 外部作用域就不能被销毁

示例

```js
function outer() {
  var a = 1

  function inner() {
    console.log(a)
  }

  return inner
}

var fn = outer()
fn()
```

输出：

```txt
1
```

为什么 a 没有销毁？

因为：

- inner 还在引用a

- JS 垃圾回收机制发现还有引用

- 所以不会回收

闭包的应用场景

1. 数据私有化

2. 模块化

3. 防抖节流

4. Vue 响应式原理

## 七、for + setTimeout 原理

```js
for (var i = 0; i < 3; i++) {
  setTimeout(() => {
    console.log(i)
  }, 0)
}
```

输出：

```txt
3
3
3
```

原因：

- setTimeout 是宏任务

- for 循环先执行完

- i 变成 3

- 回调访问的是同一个 i

let 版本：

```txt
0
1
2
```

原因：

- 每次循环创建新的块级作用域

- 每个 i 独立存在

类似于

```js
{
  let i = 0
  setTimeout(...)
}
{
  let i = 1
  setTimeout(...)
}
{
  let i = 2
  setTimeout(...)
}
```

## 八、完整知识结构图

```txt
执行上下文
 ├── 创建阶段
 │     ├── 变量提升
 │     ├── 作用域确定
 │     └── 作用域链建立
 │
 └── 执行阶段
       ├── 逐行执行
       └── 函数调用

作用域
 ├── 全局
 ├── 函数
 └── 块级

闭包
 ├── 函数 + 外部变量
 ├── 持有作用域
 └── 防止变量被销毁
```

## 面试题 1

```js
var a = 1

function test() {
  console.log(a)
  var a = 2
}

test()
```

为什么输出是 undefined？

> 函数内部的变量在创建阶段就已经存在，因此会遮蔽外部变量，打印的是内部变量的 undefined。

## 面试题 2

```js
var a = 1

function outer() {
  var a = 2
  function inner() {
    console.log(a)
  }
  return inner
}

var fn = outer()
fn()
```

输出什么？为什么？

### 🔥 正确执行流程

全局创建阶段

```js
var a = undefined
function outer() { ... }
执行阶段
a = 1
var fn = outer()
```

进入 outer

创建阶段：

```js
var a = undefined
function inner() { ... }
```

执行阶段：

```js
a = 2
return inner
```

调用 fn()

进入 inner：

它查找变量：

```txt
inner作用域
→ outer作用域
→ 全局作用域
```

在 outer 找到 a = 2

### 🎯 关键知识点

这叫：

> 词法作用域（Lexical Scope）

函数的作用域在定义时就确定了。

不是调用时确定。

## 面试题 3（判断你是否真的理解作用域链）

```js
var a = 1

function test() {
  console.log(a)
}

function test2() {
  var a = 2
  test()
}

test2()
```

输出什么？

> 1

为什么不是 2？

### 🔥核心原因

test 定义在全局。

所以它的作用域链是：

```txt
test作用域
→ 全局作用域
```

而不是：

```txt
test作用域
→ test2作用域
→ 全局
```

### 🎯 面试标准回答

> JavaScript 采用词法作用域，函数的作用域在定义时就确定，而不是调用时确定，因此 test 函数只能访问定义时所在的全局作用域。

## 进阶理解：内存角度解释闭包

```js
function outer() {
  var a = 1
  return function () {
    console.log(a)
  }
}

var fn = outer()
fn()
```

为什么 a 没被销毁？

正常情况：

> 函数执行完 → 执行上下文出栈 → 变量销毁

但这里：

> inner 还在引用 a

所以 JS 引擎会：

> 把 outer 的变量环境保存到堆内存

这就形成闭包。

## 进阶题 1

```js
function test() {
  var a = 1
  return function () {
    a++
    console.log(a)
  }
}

var fn1 = test()
fn1()
fn1()
fn1()
```

输出什么？

为什么？

### 🔥 正确底层解释（面试标准版）

当执行：

> var fn1 = test()

执行流程：

① 进入 test 执行上下文

创建变量环境：

> a = undefined

执行阶段：

> a = 1 return inner

此时：

- inner 函数引用了变量 a

- 所以 test 的变量环境不会被销毁

- 被保存在堆内存中

② 每次执行 fn1()

都会访问同一个 a

> a++

因此：

> 2 → 3 → 4

⚠ 你有一个表达错误

你说：

保存到堆栈

❌ JS 只有：

栈（执行上下文）

堆（对象、闭包环境）

闭包的变量环境存在 堆内存，不是“堆栈”。

面试中这样说：

> 函数执行完本应出栈销毁，但因为内部函数仍引用外部变量，JS 会将其变量环境保存在堆内存中，形成闭包。

## 进阶题 2

```js
function test() {
  var a = 1
  return function () {
    console.log(a)
  }
}

var fn1 = test()
var fn2 = test()

fn1()
fn2()
```

输出什么？

fn1 和 fn2 共享同一个 a 吗？

为什么？

### 为什么不共享？

每调用一次：

> test()

都会：

- 创建新的执行上下文

- 创建新的变量环境

- 生成新的 a

所以：

> fn1 → 闭包1 → a(1) fn2 → 闭包2 → a(1)

是两个独立的闭包。

🔥 面试高级表达

> 每次调用 test 都会创建新的执行上下文，因此闭包持有的是不同的词法环境，变量互不影响。

## 进阶题 3（真正考察理解深度）

```js
function test() {
  var arr = []

  for (var i = 0; i < 3; i++) {
    arr.push(function () {
      console.log(i)
    })
  }

  return arr
}

var fns = test()
fns[0]()
fns[1]()
fns[2]()
```

### 输出什么？

> 3 3 3

### 为什么？

执行流程：

① for 循环结束后

> i = 3

② 所有函数共享同一个 i

因为：

> var 是函数作用域

③ 调用时访问的都是同一个 i

所以打印：

> 3 3 3

### 如何改成 0 1 2？

方法一（推荐）：

```js
for (let i = 0; i < 3; i++) {
```

因为：

let 每次循环创建新的块级作用域

方法二（经典闭包写法）：

```js
for (var i = 0; i < 3; i++) {
  ;(function (i) {
    arr.push(function () {
      console.log(i)
    })
  })(i)
}
```

通过立即执行函数创建新的作用域。

## 终极理解测试

```js
function test() {
  var a = 10

  return function (b) {
    return a + b
  }
}

var fn = test()

console.log(fn(5))

a = 100

console.log(fn(5))
```

### 输出什么？

> 15 15

### 为什么？

🔥 标准执行流程（面试级讲法）① 执行 `var fn = test()` 进入 test 执行上下文

创建阶段：

> a = undefined

执行阶段：

```js
a = 10
return inner
```

关键点：

- inner 函数引用了变量 a

- 因此 test 的变量环境不会销毁

- 被保存到堆内存中

- 形成闭包

此时内存结构可以理解为：

```txt
fn → inner函数
        ↑
     闭包环境 { a: 10 }
```

② 执行 fn(5)

查找变量顺序：

```txt
inner作用域
→ test作用域（闭包保存）
→ 全局作用域
```

找到：

> a = 10

所以：

> 10 + 5 = 15

③ 执行 a = 100

注意：

这里修改的是：

> 全局变量 a

而闭包里的 a 是：

> test函数的变量环境中的 a

这两个不是同一个变量。

④ 再执行 fn(5)

查找顺序：

```txt
inner作用域
→ test作用域（a = 10）
```

仍然是 10。

所以：

> 15

- 函数的作用域链在定义时已经确定，inner 的外部作用域是 test，而不是全局，因此优先访问 test 的变量环境。这叫词法作用域

## 升级题

```js
function test() {
  var a = 10

  return {
    get: function () {
      console.log(a)
    },
    set: function (value) {
      a = value
    }
  }
}

var obj = test()

obj.get()
obj.set(50)
obj.get()
```

> 10 50

### 🔥 标准底层执行流程

① 执行 test()

进入 test 执行上下文：

创建阶段

> a = undefined

执行阶段

```js
a = 10
return { get, set }
```

② 为什么 a 不会销毁？

因为：

> get 和 set 都引用了变量 a

只要还有函数引用这个变量：

- 执行上下文虽然出栈

- 但变量环境会被保存到堆内存

- 形成闭包

③ 执行 obj.get()

查找顺序：

```txt
get函数作用域
→ test函数作用域（闭包环境）
→ 全局作用域
```

找到：

> a = 10

输出 10。

④ 执行 obj.set(50)

修改的是：

> test函数作用域中的 a

注意：

闭包保存的是变量引用，不是变量值。

它不是把 10 存起来。

而是保存了：

> 变量 a 的内存地址

⑤ 再执行 obj.get()

访问的还是同一个 a：

> 50

- ✅ 变量的引用（同一块内存）:所以多个方法可以共享同一个私有变量。
- ❌ 变量的值（不同内存）:不能直接修改闭包中的变量值。

### 🚀 这就是“私有变量模式”

这是 JS 早期实现私有变量的经典写法。

等你学到 Vue 响应式原理，你会发现：

> Vue2 本质就是利用闭包保存依赖
